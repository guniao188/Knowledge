# 长立-外贸订单CRM方案答复V1.0

## 一、总体说明

首先感谢贵司提供了非常详细的业务需求梳理。我们理解本次系统建设的核心目标,是把**报价、议价、合同、履约**四段业务串成一条完整、可追溯的数据链路,让销售、审批、跟单、计划、采购、生产、开发、财务共用同一份数据,彻底解决**报价历史断层、价格依据丢失、合同执行靠线下表格、图片凭证无法追溯**这四大痛点。

Odoo 在**销售报价 / 订单 / 合同 / 发货 / 收款**主链路上已有较为成熟的原生支撑,同时留出了充分的二开与配置空间。以下针对贵司提出的每一条需求,我们逐条给出**当前 Odoo 是否已具备、如何具备、需要开发的边界与前置确认项**,便于贵司评估。

**图例说明**

| ------- | ------------------------ |
| ✅ 原生支持  | Odoo 标准功能开箱即用,无需开发       |
| 🔧 需配置  | 通过参数、模板、权限或字段扩展可实现,不涉及代码 |

## 二、系统定位

本系统不是普通 CRM,而是以**外贸客户询价、历史价格、报价审批、议价确认、合同履约与跨部门协同**为核心的**订单管理型 CRM**。每一次报价、议价、成交、发货、回款和沟通资料,都会沉淀到 **客户 / 客户号码 / 长立号码 / 报价单号 / 合同号** 五个维度上,实现"任何时候都能追溯到当初为什么这样报、这样成交"的目标。

## 七、合同管理

### 7.1 合同定义

合同的定义需要贵司先与我们对齐。**Odoo 原生认为"订单确认后即为合同"**,基于此,我们建议以下两条实现路径,请贵司结合内部管理习惯选择:

| 方案      | 思路                                | 优点                         | 缺点                        |
| ------- | --------------------------------- | -------------------------- | ------------------------- |
| **方案一** | 用 Odoo 原生:报价 → 议价 → 订单确认 = 合同     | 开发量小、上线快、维护简单              | 需要贵司接受"订单即合同"的语义          |
| **方案二** | 在订单之上增补独立的【合同】模块,支持从多张报价单挑明细组合成合同 | 完全贴合"合同是独立单据"的业务语义,支持多来源合并 | 开发量较大,涉及打印、审批、跨部门联动等一系列配套 |

### 7.2 合同生成方式

贵司提出的四种合同生成路径:

1. 从最近报价表中选择部分产品生成合同
   
2. 从整张报价单生成合同 ：报价单增加按钮，转成订单
3. 从多个报价单中选部分产品组合成合同
4. 从最终议价记录生成合同
5. 无长立号码产品在生成合同时补充或标记待补充

 默认格式为客户号/数量，客户号/数量。取最近的报价的价格进行匹配。
从客户页面增加一个按钮【生成复购订单】，点击后显示弹窗，按照规则复制进去点击生成报价按钮，如果未匹配的客户编号报错提醒。规则的默认格式为客户号/数量，客户号/数量。取最近的报价的价格进行匹配。

报价单明细界面也可以批量选择创建报价单。
 报价单明细生成订单增加限制：同一个客户、同一个客户货号，如果不是最新的报价，不允许生成订单。进行报错
 订单明细需要显示最近报价日期，为报价最新的审批的日期。显示的日期如果超过6个月标红，超过3个月显示黄色。自动任务每天检查是否超期。
 
**回应**:

- **方案一**(原生):在报价 / 议价环节直接调整明细行,提交审批后订单确认即为合同。**多来源合并需要人工拆合。**
- **方案二**(二开):新增【生成合同】按钮,弹窗中勾选来源报价单和明细行,自动合成合同。此为独立开发任务。
**演示会议后初步方案：**
	在销售订单明细增加按钮，点击跳出弹窗，在弹窗中支持按照产品的编号、oe号、客户编号搜索，然后选择需要加入明细行中。

## 八、合同拆分、合并与发货

- 合同执行以**合同产品行 + 数量**为最小管理单位,拆分、合并与发货必须从原合同行选择,系统不允许超出未发数量。
- **现状**:
  - 我们已经在贵司环境安装了**订单拆分 / 合并插件**,可满足"从原合同行组合出新合同"的需求;
  - **发货、收款**均由最终合同(销售订单)触发,业务闭环清晰;
  - **发货单本身的拆分与合并**属于 Odoo 原生能力;
  - Odoo 未内置"发货数量 ≤ 合同未发数量"的校验,如需强制卡控,建议做**开发限制**,避免超发。

| #   | 场景               | 现状                  | 方案    | 最终方案 |
| --- | ---------------- | ------------------- | ----- | ---- |
| 1   | 合同拆分 / 合并        | 已安装插件覆盖             | ✅ 已就绪 |      |
| 2   | 发货单拆分 / 合并       | Odoo 原生             | ✅ 原生  |      |
| 3   | 发货 / 收款触发        | 由最终销售订单触发           | ✅ 原生  |      |
| 4   | 发货数量 ≤ 合同数量的强制校验 | 原生未卡控,建议二开加"未发数量"校验 | 🛠 二开 |      |
|     |                  |                     |       |      |
